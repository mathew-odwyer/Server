# Winterhaven - MVP Specification Document

## Summary

Winterhaven is a 2D top-down online role-playing game showcasing expertise in distributed systems, real-time networking, secure authentication, and multiplayer game architecture. This MVP demonstrates production-ready skills in containerized microservices, database design, and real-time client-server communication.

**GOAL:** A functional multiplayer RPG with secure authentication, real-time movement, dynamic map loading, and interactive chat systems.

---

## Technical Architecture

| Component | Technology | Purpose |
|-----------|-----------|---------|
| Client | GameMaker (GML) | Game rendering, input handling, client-side prediction |
| Game Server | GameMaker (GML) | Real-time game logic, state authority, world simulation |
| Web API | ASP.NET Core, C# | Authentication, user management, data persistence |
| Database | MSSQL + EF Core | Player data, accounts, persistent storage |
| Reverse Proxy | Caddy | SSL/TLS termination, routing |
| Orchestration | Docker Compose | Container management, service coordination |

### 2. Dynamic Map Loading System
**Priority:** CRITICAL  
**Estimated Effort:** High

#### Requirements

##### Initial Map Load (Login)
- [ ] **Server-Side Map Detection**
  - [ ] Parse Tiled .world file on server startup
  - [ ] Index maps by region coordinates
  - [ ] Determine player's spawn region from stored X, Y coordinates
  - [ ] Load appropriate region map on player login

- [ ] **Map Data Transmission**
  - [ ] Send map data to client during login sequence
  - [ ] Client renders received map data

##### Dynamic Region Transitions
- [ ] **Region Boundary Detection**
  - [ ] Server monitors player position
  - [ ] Detect when player crosses region boundaries
  - [ ] Load adjacent region

- [ ] **Adjacent Region Loading**
  - [ ] Load new region map if not already loaded
  - [ ] Send new map data to client
  - [ ] Client transitions between regions smoothly
    - [ ] Do this via fade in/fade out effect.

##### Interior/Building System
- [ ] **Interior Entry/Exit**
  - [ ] Define entrance/exit collision object
  - [ ] Load interior map on entry
    - [ ] Render collision tiles in differnt color for easy debugging
  - [ ] Teleport player to interior coordinates
  - [ ] Return to overworld on exit
  - [ ] Going to interior should not change "region" for player (important for chat)

##### Map Memory Management
- [ ] **Automatic Map Unloading**
  - [ ] Unload maps when no players in map/region
  - [ ] Optional: Delay unload by 30-60 seconds for quick returns

---

### 3. Chat System
**Priority:** HIGH  
**Estimated Effort:** Medium

#### Requirements

- [ ] **World Chat**
  - [ ] Broadcast messages to all players in world instance
  - [ ] Message format: `[World] PlayerName: message`
  - [ ] Server validates and sanitizes input

- [ ] **Local Chat**
  - [ ] Send messages only to players in same region
  - [ ] Message format: `[Local] PlayerName: message`
  - [ ] Different visual styling from world chat

- [ ] **Private/Whisper Chat**
  - [ ] One-to-one messaging between players
  - [ ] Message format: `[From PlayerName]: message` / `[To PlayerName]: message`
  - [ ] Server routes to specific player
  - [ ] Error handling for offline/invalid recipients

- [ ] **Chat UI**
  - [ ] Toggle between chat channels
  - [ ] Visual distinction between chat types

#### Technical Notes
- Command structure: `/w PlayerName message`, `/l message`, `/w message`
- Maximum message length, `string_lower`.
- Basic profanity filter (optional for MVP)
